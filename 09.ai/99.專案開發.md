### 🏗️ 三維度治理模型 (區分「工具」、「法律」與「專業」)

| 分類名稱                                | 包含檔案                                       | 管理目標                                                              | AI 的角色感知                        |
| :-------------------------------------- | :--------------------------------------------- | :-------------------------------------------------------------------- | :----------------------------------- |
| **1. 工具 (Tools)**                     | `README.md`, `TASK.md`, `.env`, `.eslintrc`    | **環境與執行**：定義專案環境、如何啟動與當前任務進度。                | 這是我的「工具箱」與「工作看板」。   |
| **2. 法律 (Law)**                       | `GUIDELINES.md`, `ARCHITECT.md`, `ADR.md`      | **行為與架構約束**：定義 AI 的思考邏輯、技術架構選擇與決策記錄。      | 這是我的「法律」與「憲法」。         |
| **3. 專業 (Professional)**              | `CUSTOM_SKILL.md` (或 SOP 邏輯檔案)            | **業務專長與 SOP**：針對特定業務邏輯 (如校務系統) 的深度 Know-how。   | 這是我的「專業證照」與「作業手冊」。 |

---

### 💡 為什麼要這樣分類？ (優化效益)

1.  **解決「README 肥大化」問題**：
    - 傳統開發中，README 會塞滿安裝步驟、技術規範、業務邏輯。
    - **優化後**：README 回歸本質（如何啟動專案），而 AI 的各種高頻約束被抽離到法典中，避免 AI 在讀取 README 時被過多細節干擾，導致忽略了核心環境設定。

2.  **實現「跨專案複用」的高效率**：
    - **工具層 (Tools)** 隨框架與環境變動。
    - **法律層 (Law)** 在您同一個團隊內基本上是「通用的」（例如您的 SQL 規範）。
    - **專業層 (Professional)** 則是可以打包的「知識包」，例如您有一個 `SCHOOL_EXPERT.md`，可以輕鬆在不同校務子專案中掛載。

3.  **精確的對策 (Context Injection)**：
    - 當 AI 只是要修一個 Bug 時，您只需要餵給它「工具」與「法律」。
    - 當您要開發新業務功能時，才需要餵給它「專業」。這能大幅減少 AI 的 Token 消耗並提升準確率。

### 📂 物理結構優化建議

如果您想進一步實現物理上的區分，我建議在專案中建立一個 `.ai/` 或 `docs/governance/` 資料夾，但保留核心文件夾在根目錄以供工具掃描：

```text
/             (根目錄)
├── .env, .gitignore, .eslintrc (工具/基礎)
├── README.md                   (工具/簡介)
├── TASK.md                     (工具/當前進度)
└── .ai/      (AI 治理目錄)
    ├── guidelines/
    │   ├── GUIDELINES.md       (法律: 行為規範)
    │   ├── ARCHITECT.md        (法律: 架構規範)
    │   └── ADR.md              (法律: 決策紀錄)
    └── skills/
        └── CUSTOM_SKILL.md     (專業: 領域智慧)
```

### 🛠️ 實際範例檔案 (以校務系統為例)

根據「物理結構優化建議」，以下是該校務系統專案在各個維度中的實際檔案配置：

#### 1. [工具] 根目錄看板：`TASK.md`
用於追蹤當前開發任務的進度。

```markdown
# 任務：開發「單筆成績修改」功能與日誌
- [x] 在 `GradeController` 增加 `Edit` Action (GET)
- [/] 實作 SQL 修改邏輯：
    - 更新 `STUDENT_GRADE` 分數
    - 關聯 `SYS_USER` 紀錄修改者
- [ ] 實作修改日誌：寫入 `SYS_LOG` 表格
- [ ] 增加「審核狀態」檢核：已鎖定學期禁止修改
```

#### 2. [法律] 治理目錄：`.ai/guidelines/GUIDELINES.md`
定義放諸四海皆準的靜態規範與法典。

```markdown
# 校務系統開發法典 (GUIDELINES)

## 🗄️ SQL 規範 (MANDATORY)
*   欄位名稱：全大寫並依據 MCP 檢核結果 (如 `STU_ID`, `SCORE`)。
*   保留字：全小寫 (如 `select`, `from`, `left join`)。
*   關聯：優先使用 `left join` 以確保資料完整性。
*   安全：必須使用參數化查詢，禁止拼接 SQL 字串。

## ✒️ 命名規範
*   後端 C#：PascalCase (如 `CalculateTermGpa`)。
*   前端 JS：camelCase (如 `studentDataTable`)。
*   View：對應 Action 名稱 (如 `Edit.cshtml`)。
```

#### 3. [專業] 智慧目錄：`.ai/skills/SCHOOL_EXPERT.md`
記載特定業務領域的 Know-how 與 SOP。

```markdown
# 校務業務領域專家 (SCHOOL_EXPERT)

## 🎓 業務核心 SOP
### 1. 學分計算邏輯
*   必修課加權系數：1.5；選修課：1.0。
*   畢業門檻：總學分必須高於 128，且包含核心通識 12 學分。

### 2. 擋修規則 (Prerequisites)
*   若要修習 `PROG_020` (進階程式)，必須先取得 `PROG_010` 學分且成績高於 70。
*   如果是轉學生，則改檢核 `CREDIT_TRANSFER` 表格中的 `PROG_BASIC` 標記。
```

#### 4. [工具] 基礎配置：`.env`
環境變數與敏感資訊。

```bash
# 資料庫連接字串
DB_CONNECTION="Server=school-db-prod;Database=STUDENT_DB;User Id=sa;Password=********;"
# 環境標籤
ENVIRONMENT="Development"
```
