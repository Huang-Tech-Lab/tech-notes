# 法律規範：法律顧問 (Legal Advisor) 操作手冊

本文件詳述專案中的「法律顧問」角色，定義其管理的檔案範圍、執掌權力，以及在一套專案從無到有、或舊專案治理化過程中的標準操作流程 (SOP)。

---

## ⚖️ 角色定義：法律顧問 (Legal Advisor)

法律顧問的核心目標是確保專案具備**「可預測性」**。他透過維護一套「靜態規範」，將開發者的模糊意圖轉化為 AI 可嚴格遵守的契約。

### 1. 職責範圍 (Legal Scope)
*   **管法律**：定義專案的基本原則、命名規範與代碼哲學。
*   **管環境**：確保技術邊界（如 Node 版本、Lint 規則、環境變數）被機器嚴格遵守。
*   **管記憶**：透過歷史決策記錄 (ADR) 與變更日誌 (Changelog)，確保 AI 不會重啟對話後就忘記之前的決策。

### 2. 治理檔案清單

| 分類 | 關鍵檔案 | 職能說明 |
| :--- | :--- | :--- |
| **外部法律 (Human-Readable)** | `README.md`, `GUIDELINES.md`, `ARCHITECT.md` | 定義專案「是什麼」、「怎麼做」與「架構長什麼樣」。 |
| **歷史判規 (Memory)** | `ADR.md`, `CHANGELOG.md` | 定義「為什麼這樣做」與「過去做了什麼」。 |
| **機器法律 (Machine-Readable)** | `.env`, `.gitignore`, `.editorconfig` | AI 在執行任何指令前必須遵守的硬性邊界。 |
| **技術約束 (Code Contracts)** | `package.json`, `.eslintrc`, `.tsconfig` | 定義語言、框架、庫的版本與靜態檢查規則。 |

---

## 📂 治理檔案範例 (Templates & Examples)

法律顧問應確保以下檔案的結構化與清晰度。以下提供標準範例供參考或複製：

### 1. GUIDELINES.md (開發指南)
> **作用**：定義代碼風格與開發邊界。

```markdown
# 專案開發指南

## 1. 命名規範
*   **變數/函式**：使用 `camelCase` (例如: `getUserData`)。
*   **檔名**：使用 `kebab-case` (例如: `user-profile.controller.ts`)。
*   **常數**：使用 `SCREAMING_SNAKE_CASE` (例如: `MAX_RETRY_COUNT`)。

## 2. 錯誤處理
*   禁止使用空的 `catch` 區塊。
*   所有 API 錯誤必須包含 `code` 與 `message`。

## 3. Git 指令
*   提交格式：`feat: <內容>`, `fix: <內容>`, `docs: <內容>`。
```

### 2. ADR.md (架構決策紀錄)
> **作用**：記錄「為什麼」這麼做的技術決策歷史。

```markdown
# ADR 001: 選擇 Tailwind CSS 作為樣式方案

## 狀態
已通過 (Accepted)

## 背景
專案需要快速迭代且具備高度客製化的 UI，傳統 CSS 模組化開發速度較慢。

## 決策
決定使用 Tailwind CSS v4。

## 後果
*   優點：提升開發速度，減少 CSS 檔案大小。
*   缺點：HTML 類名會變長，初學者有學習曲線。
```

### 3. ARCHITECT.md (架構說明)
> **作用**：定義專案結構與數據流向。

```markdown
# 系統架構圖 (C4 Model)

## 目錄結構
*   `/src/core`: 核心邏輯 (與框架無關)。
*   `/src/infra`: 基礎設施 (DB, API Clients)。
*   `/src/ui`: 使用者介面 (Components, Pages)。

## 數據流
`UI -> Action -> UseCase -> Repository -> Database`
```

### 4. CHANGELOG.md (變更日誌)
> **作用**：記錄各版本的具體異動。

```markdown
# 變更日誌

## [1.1.0] - 2026-04-19
### Added
- 新增法律顧問 (Legal Advisor) 治理框架。
- 建立 `03.法律規範.md` 操作手冊。

### Fixed
- 修正 `Antigrivaty.md` 中的跳轉連結失效問題。
```

### 5. .env.example (環境變數範例)
> **作用**：宣告必要的變數框架，嚴禁包含真實金鑰。

```bash
# API 服務
API_RETRY_LIMIT=3
DB_CONNECTION_TIMEOUT=5000

# 安全金鑰 (請由開發者本機 .env 自行填寫)
SECRET_TOKEN=your_placeholder_here
```

### 6. .editorconfig (編譯器約束)
> **作用**：強制執行跨編輯器的一致縮排。

```ini
root = true

[*]
indent_style = space
indent_size = 4
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true
```

## 🛠️ 新專案：如何初始化法律規範

當您啟動一個全新的空白專案時，應遵循以下「立法」流程：

### 第一階段：基礎法治建設
1.  **建立 `.env.example`**：宣告專案運作所需的環境變量框架。
2.  **初始化 `.gitignore`**：防止敏感資訊外洩。
3.  **建立 `README.md`**：定義專案目標與開發環境啟動方式。

### 第二階段：Agent 治理化
要求 AI 執行以下 Prompt 以生成治理雛形：
> 「根據目前的技術堆疊（如 Next.js / Python），請為我生成初始的 `GUIDELINES.md` 與 `ARCHITECT.md`。重點包含：命名風格、目錄結構規範、以及錯誤處理原則。」

### 第三階段：開啟 ADR 維護
建立一個 `09.ai/ADR.md` 檔案，記錄下第一個重大決策（例如：為什麼選擇這套 UI Kit）。

## ♻️ 舊專案：如何導入法律顧問 (治理化 SOP)

對於已經在運行但缺乏規範的「野生專案」，法律顧問應進行以下「法律審計」：

### 步驟 1：法律巡邏 (Audit)
要求 AI 執行審計任务：
> 「請讀取專案的所有目錄結構與 `package.json`，分析目前的開發慣例，並產出一份『現況評估報告』，指出目前的規範缺口。」

### 步驟 2：法律追認 (Ratification)
根據審計結果，要求 AI 將「既成事實」轉化為「成文法」：
> 「基於剛才的審計，請將我們目前的代碼風格與目錄結構整理成 `GUIDELINES.md`，作為未來的法律藍本。」

### 步驟 3：修復技術約束
檢查 `.gitignore` 與 `.editorconfig` 是否存在。若缺失，要求 AI 根據現有環境補齊，以達到「產出代碼風格一致」的目的。

## 🔄 法律顧問的日常維護

1.  **ADR 固定更新**：每當發生重大架構變更（如：更換資料庫、新增認證模式），法律顧問必須在 `ADR.md` 留下判決書。
2.  **Changelog 自動化**：開發結束後，利用 `gh` 與 Commit 紀錄，要求 AI 自動更新 `09.ai/CHANGELOG.md`。
3.  **定期法律覆核**：每個月執行一次 `/stats`，檢查 Context 佔用是否過高，並由法律顧問評估是否需要更新 `GUIDELINES.md` 以精簡規則。
