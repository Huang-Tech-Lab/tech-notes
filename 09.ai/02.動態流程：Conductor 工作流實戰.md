## 總經理的角色職責

總經理 (Conductor) 負責處理專案的「生命力」。它不負責寫法律，但它負責確保法律被落地執行。其核心價值在於將模糊的需求，轉化為可以精確追蹤的任務軌跡 (Tracks)。

### 為什麼我們需要動態流程？
*   **防止進度迷失**：AI Agent 在進行複雜任務時，容易「做 A 忘 B」。Conductor 強制記錄每一步的進度至 `plan.md` 與 `task.md`。
*   **支援多人協作**：透過不同的 Track 軌道，多人（或多個 Agent）可以互不干擾地開發。
*   **自動化歸檔**：任務完成後，所有執行細節、錄影、Logs 都會被完整保存。

## 動態工作流：生命週期

一個標準的「動態任務」會經歷以下五個階段：

1.  **啟動 (Start)**：使用 `/conductor:new_track` 定義目標（總經理入場）。
2.  **擬定計畫 (Plan)**：總經理生成 `plan.md`。此時 Agent 會**主動諮詢靜態規範 (Consulting Legal)** 以確保計畫合規。
3.  **執行 (Execute)**：使用 `/conductor:implement` 執行任務，過程中會頻繁更新 `task.md`。
4.  **品質驗證 (QA)**：由品管主管 (TDD) 進行測試。
5.  **關閉 (Close)**：任務完成，軌跡歸署，法律顧問若有必要則更新 ADR。

## 多人協作實務 (Multi-Agent Sync)

在 Conductor 框架下，多人協作的核心在於 **「軌道隔離 (Track Isolation)」**：

- **一人一軋 (One Agent, One Track)**：不同的團隊成員應在各自的 Track 中工作。
- **Git 狀態同步**：Conductor 執行前會檢查 git 狀態，確保你的「動態動作」是有版控保護的。
- **衝突處理**：若多個 Track 修改到同一個靜態規範檔案，必須由「法律顧問」進行人手審核核准。
