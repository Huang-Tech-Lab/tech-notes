## 專案的兩大支柱：動態流程 vs. 靜態規範

這套架構將 AI 開發拆解為兩個互補的維度：一個負責「守穩（規範）」，另一個負責「進攻（執行）」。

### ⚖️ 靜態規範 (Static Governance)
這是專案的「法律」與「邊界」，定義了開發中的 **「不變量」**。它是 AI 的「離線參考手冊」，確保無論 Agent 如何更替，開發標準始終如一。

| 分類 | 代表角色 | 關鍵檔案 | 核心職責 |
| :--- | :--- | :--- | :--- |
| **管理法律** | ⚖️ 法律顧問 | `README.md`, `GUIDELINES.md`, `ARCHITECT.md`, `ADR.md` | 定義專案願景、通用規範與重大架構決策。 |
| **技術規範** | ⚖️ 法律顧問 | `.env`, `.gitignore`, `package.json`, `.eslintrc` | 提供機器可讀的環境約束、依賴管理與代碼檢查規則。 |
| **領域專業** | 📜 領域專家 | `CUSTOM_SKILL.md` | 針對特定領域（如 AI 治理、後端架構）的專業行為 SOP。 |
| **美學設計** | 🎨 視覺總監 | `DESIGN.md` | 維護視覺一致性與元件庫的使用規範。 |

### 🚀 動態流程 (Dynamic Workflow)
這是專案的「生命力」與「進度」。它處理開發中的 **「變量」**，負責將模糊的 User Request 轉化為可量化、可驗證的任務軌跡。
*   **核心價值**：確保複雜任務不迷失，透過進度追蹤（TASK.md）實現透明度。
*   **關鍵思維**：它是 AI 的「行動導覽員」。

### 核心對照表

| 維度 | 靜態規範 (Governance) | 動態流程 (Execution) |
| :--- | :--- | :--- |
| **比喻** | 國家的法律、公司的規章 | 專案的進度表、執行軌道 |
| **主要檔案** | `GUIDELINES.md`, `ARCHITECT.md` | `plan.md`, `TASK.md` |
| **存續時間** | 長期穩定，隨架構迭代更新 | 短期存在，隨任務啟動與關閉 |
| **核心角色** | 法律顧問、領域專家、視覺總監 | 總經理 (Conductor)、品管主管 (TDD) |
| **交互關係** | 指導動態流程的「合法性」 | 落實靜態規範的「執行力」 |

## 方案對照：輕量級 vs. 專業級

在使用 AI 開發時，根據需求複雜度，我們可以選擇不同的執行路徑：

| 方案 | 核心工具 | 適合情境 | 運作特點 |
| :--- | :--- | :--- | :--- |
| **🚀 輕量化 (Lightweight)** | `task.md` | 小型專案、單人開發、短期腳本 | **「開箱即用」**。直接在根目錄建立任務清單，AI 根據清單逐項執行並同步狀態，適合快速迭代。 |
| **💼 專業級 (Professional)** | `Conductor` 框架 | 複雜架構、團隊協作、長期維護 | **「軌道管理」**。透過軌道 (Tracks) 隔離不同功能，具備完整的啟動、計畫、執行與歸檔流程，適合高穩定性需求。 |

## 五大核心角色定義

本專案將 AI Agent 視為專業團隊成員，依據其「管理維度」與「互動頻率」劃分為五大核心角色：

### 💼 [總經理 (Conductor)](02.動態流程：Conductor 工作流實戰.md)
*   **檔案**：`.conductor/`, `TASK.md`, `plan.md`
*   **職責**：管「進度」與「目標」。將需求拆解為任務軌道、分配工作，是動態流程的發起者。

### 🧪 [品管主管 (TDD)](TEST_GUIDELINES.md)
*   **檔案**：`tests/`, `TEST_GUIDELINES.md`
*   **職責**：管「正確性」。透過自動化測試驗證代碼是否符合預期，確保系統穩健。

### ⚖️ [法律顧問 (Legal Advisor)](03.法律規範.md)
*   **檔案**：`README.md`, `GUIDELINES.md`, `ARCHITECT.md`, `ADR.md`, `CHANGELOG.md`, `.env`, `.gitignore`
*   **職責**：管「法律」、「技術環境」與「記憶」。提供所有人類與機器可讀的約束規則。

### 📜 [各領域專家 (Specialized Expert)](04.各領域專家.md)
*   **檔案**：`CUSTOM_SKILL.md`
*   **職責**：管「專業行為」。為特定的複雜領域（如 ASP.NET MVC, AWS）提供標準操作 SOP 與避雷指南。

### 🎨 [視覺總監 (Design System)](05.視覺總監.md)
*   **檔案**：`DESIGN.md`
*   **職責**：管「一致性」與「美感」。確保 UI 組件（Bootstrap, DevExtreme 等）符合系統美學調性。

## 交互運作流程：當動態遇上靜態

兩者的結合不是一種束縛，而是一套 **「開發防呆機制」**。典型的運作路徑如下：

1.  **情境諮詢 (Legal Briefing)**：當 Agent 接到任務時，第一步不是寫 code，而是「讀法」。這包括 `GUIDELINES.md` 與專屬的 `CUSTOM_SKILL.md`。
2.  **計畫定錨 (Plan Anchoring)**：總經理（Conductor）在撰寫 `implementation_plan.md` 時，必須將「靜態約束」納入考量。例如：是否符合目錄結構？是否使用了正確的命名規範？
3.  **合規執行 (Compliant Execution)**：在執行軌道（Track）中，品管主管（TDD）會根據技術法律（如測試指南）進行驗證。
4.  **經驗沉澱 (Static Feedback Loop)**：任務完成後，若有「值得被留存的決定」，我們會將其記錄到 `ADR.md` 或更新規範檔案。此時，今日的「動態結果」便成為了明日的「靜態財富」。

> [!TIP]
> **黃金法則**：當你不知道該做什麼時，看【動態流程】（TASK.md）；當你不知道該怎麼做才對時，看【靜態規範】（GUIDELINES.md）。
