## 💼 1. 總經理的角色職責

總經理 (Conductor) 負責處理專案的「生命力」。它不負責寫法律，但它負責確保法律被落地執行。其核心價值在於將模糊的需求，轉化為可以精確追蹤的任務軌跡 (Tracks)。

### 為什麼我們需要動態流程？
*   **防止進度迷失**：AI Agent 在進行複雜任務時，容易「做 A 忘 B」。Conductor 強制記錄每一步的進度至 `plan.md` 與 `task.md`。
*   **支援多人協作**：透過不同的 Track 軌道，多人（或多個 Agent）可以互不干擾地開發。
*   **自動化歸檔**：任務完成後，所有執行細節、錄影、Logs 都會被完整保存。

## 🔄 2. 動態工作流：生命週期

一個標準的「動態任務」會經歷以下五個階段，並伴隨對應的系統產出：

1.  **啟動 (Start)**：使用 `/conductor:new_track` 定義目標。
    *   📄 **生成檔案**：`.conductor/tracks/track-xxx.md` (定義規格與初始軌跡)。
2.  **擬定計畫 (Plan)**：總經理生成計畫。此時 AI 會諮詢靜態規範以確保合規。
    *   📄 **生成檔案**：`implementation_plan.md` (詳細執行路徑、影響檔案與步驟)。
3.  **執行 (Execute)**：使用 `/conductor:implement` 執行任務。
    *   📄 **生成檔案**：`task.md` (動態分級任務清單) 與實際產出的 **功能程式碼 (Source Code)**。
4.  **品質驗證 (QA)**：由品管主管 (TDD) 進行測試。
    *   📄 **生成檔案**：`tests/` 目錄下的測試案例、`test_report.md` 或 `bug_fix.md`。
5.  **關閉 (Close)**：任務完成，軌跡歸置。
    *   📄 **生成檔案**：`walkthrough.md` (成果演示與錄影連結)；若有架構決策，同步更新 `09.ai/ADR.md`。

## 💡 治理建議：雙軌制選擇

在專業級專案中，我們鼓勵使用 Conductor，但如果是「文件內容」的修改，建議採取以下雙軌平衡：

| 修改類型 | 建議模式 | 示例 |
| :--- | :--- | :--- |
| **微調 (Minor Tweak)** | **對話式修改** | 改錯字、調整一句話語氣、修改標題。直接對 AI 說：「這裡改一下」。 |
| **任務 (Task)** | **輕量級流程 (TASK.md)** | 撰寫一個新的子段落。在 `TASK.md` 標註後由 AI 批次完成。 |
| **工程 (Project)** | **Conductor 指令** | 結構重組、文件庫風格統化、建立全新模組的文件體系。 |

## 🛠️ 3. Conductor 指令全集

| 指令 | 作用 | 使用時機 | 執行結果 |
| :--- | :--- | :--- | :--- |
| `/conductor:setup` | 在新專案或環境中初始化 Conductor 配置。 | 第一次使用 Conductor 時。 | 建立 `conductor.json`、`bin/conductor-setup` 等腳本，配置工作空間。 |
| `/conductor:new_track` | 建立新軌跡 | 當您要開始一個新的功能開發、重構或 Bug 修正時。 | 建立 Track 目錄，生成 `spec.md` 與 `plan.md` 初始文件，並記錄於 `metadata.json`。 |
| `/conductor:status` | 查看進度 | 隨時查看當前所有軌跡的進度、當前 Active Track 與下一步建議。 | 產出所有 Tracks 的狀態矩陣（包含完成度 %、狀態與負責 Agent）。 |
| `/conductor:implement` | 自動執行 | 已完成計畫審閱，準備讓 AI 開始自動化執行代碼實作時。 | 執行 `/plan.md` 中的 Task，自動修改代碼、執行測試並產生連續的 Git commit。 |
| `/conductor:revert` | 邏輯式撤銷 | 當計畫執行出錯或需要回滾到特定階段時。 | Git 將專案回滾至特定 Task/Phase/Track 的 commit 點。 |
| `/conductor:manage` | 軌跡生命週期管理 | 需要封存已完成任務、清理舊分身或重新命名 Track 時。 | 更新 `tracks/` 目錄結構，封存檔案至 `archive/`。 |
| `/conductor:validator` | 專案健康檢查 | 在 `/conductor:implement` 之後，確認代碼與 Spec 是否完全契合。 | 產出驗證報告，列出契合度評分及潛在的缺口。 |

## 👥 4. 多人協作實務 (Multi-Agent Sync)

在 Conductor 框架下，多人協作的核心在於 **「軌道隔離 (Track Isolation)」**：

*   **一人一軌 (One Agent, One Track)**：不同的團隊成員應在各自的 Track 中工作。
*   **Git 狀態同步**：Conductor 執行前會檢查 git 狀態，確保你的「動態動作」是有版控保護的。
*   **衝突處理**：若多個 Track 修改到同一個靜態規範檔案，必須由「法律顧問」進行人工審核核准。

## 🎓 5. 實務練習：文件寫作與優化

如果您想練習 Conductor 工作流，最直覺的方式就是 **「使用 Conductor 來撰寫與調整本專案的文件」**。

### 練習情境：在 `01.輕量級專案.md` 新增「常見問題 (FAQ)」章節

#### 🛠️ 步驟 0：環境初始化 (Setup) - *僅限初次使用*
> **指令**：`/conductor:setup`
> **作用**：確保您的專案目錄具備 Conductor 所需的 `conductor.json` 與基礎結構。如果您是第一次在練習本專案，請務必先執行此指令。

#### 🔔 步驟 1：啟動任務 (New Track)
> **指令**：`/conductor:new_track "優化輕量級專案文件：新增 FAQ 章節"`
> **練習要點**：
> *   AI 會掃描專案並識別這是一個文件撰寫任務。
> *   AI 會詢問：FAQ 要包含哪些問題？
> *   您可以回答：「包含：1. 為什麼要用 TASK.md 而不直接做？ 2. 什麼時候該合併檔案？」

#### 📝 步驟 2：生成規格與計畫 (Spec & Plan)
> **觀察產出**：
> *   在 `conductor/tracks/optimize-lightweight-doc_YYYYMMDD/` 下查看：
>     *   `spec.md`：確認寫作範圍與 FAQ 的具體題目。
>     *   `plan.md`：步驟會包含「準備 FAQ 內容」、「插入 Markdown」、「更新目錄」與「格式校對」。

#### 🚀 步驟 3：執行開發 (Implement)
> **指令**：`/conductor:implement {Track_ID}`
> **關鍵行為**：AI 會開始根據計畫自動執行。您會看到它自動修改文件，並在每次修改後 commit，記錄下您的思考與「寫作軌跡」。

#### 📊 步驟 4：查看進度 (Status)
> **指令**：`/conductor:status`
> **練習要點**：確認任務完成百分比，感受 Conductor 如何讓即使是「文字工作」也能像代碼開發一樣被精確管理。
